# 40 — 테스트

## 완료의 정의
- 버그 수정은 회귀 테스트(수정 전엔 실패, 수정 후엔 통과)가 추가될 때까지 끝난 게 아니다.
- 새 기능은 해피패스 1개와 엣지케이스/실패 경로 1개 이상의 테스트가 있어야 끝난다.

## 테스트할 것
- 공개 API의 행동(내부 구현 디테일이 아니라).
- 엣지 케이스: 빈 입력, 최댓값, 경계값, 유니코드, 타임존, 동시 호출, 네트워크 실패.
- 에러 경로: 모든 `raise`/`throw`/`return Err(...)`은 정말 도달 불가하지 않은 한 테스트가 있어야 한다.

## 테스트하지 말 것
- 서드파티 라이브러리 내부.
- 공개 테스트로 이미 커버되는 비공개 헬퍼를 따로 테스트.
- 자동 생성 코드.

## 테스트 구조
- 한 개념당 한 어서션(같은 개념에 여러 `assert` 줄은 OK).
- Arrange / Act / Assert 배치.
- 이름은 `test_<단위>_<시나리오>_<기대결과>` 또는 프로젝트의 로컬 관례.

## 테스트 실행
- 작업 완료를 선언하기 전에 프로젝트 테스트 명령을 돌린다.
- 스위트가 느리면 타깃 파일/테스트를 먼저, 그 다음 전체.
- 플래키 테스트는 **CI를 초록으로 만들려고 스킵하지 않는다.** 원인을 진단하고 고치거나 사용자에게 알린다.

## 커버리지
- 숫자가 아니라 의미 있는 커버리지를 본다. 새 코드는 머지 전에 테스트가 있어야 한다.
