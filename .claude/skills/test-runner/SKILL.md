# 테스트 러너 스킬

## 1. 테스트 명령 감지

`Makefile`, `package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`,
`Gemfile`, `pom.xml`, `build.gradle`을 차례로 본다. 발견한 테스트 명령:

- `make test`
- `npm test` / `pnpm test` / `yarn test`
- `pytest` / `python -m pytest` / `uv run pytest`
- `cargo test`
- `go test ./...`
- `bundle exec rspec`
- `mvn test`
- `./gradlew test`

`README.md`나 `CONTRIBUTING.md`에 다른 명령이 있으면 그것을 우선한다.

## 2. 좁게 시작해서 넓힌다

먼저 변경된 영역에 해당하는 테스트만 돌린다. 통과하면 그제야 전체 스위트.
이렇게 하면 무관한 실패에 시간을 낭비하지 않는다.

## 3. 실패 분석

실패가 나면 절대 무시하지 말고:
1. 첫 번째 실패의 전체 출력을 읽는다 (잘라내지 말 것).
2. 실패가 *현재 변경*에 의한 것인지 *기존 깨진 상태*인지 판별한다(`git stash`로 비교).
3. 실패가 진짜 버그인지 테스트 자체의 버그인지 판별한다.
4. 진짜 버그면 → `debug-loop` 스킬.
5. 테스트 버그면 → 테스트를 고치되, 어서션을 약화해서 통과시키지 않는다.

## 4. 누락된 테스트 작성

- 신규 코드: 해피패스 1개 + 엣지케이스 1개 이상.
- 버그 수정: 회귀 테스트(수정 전엔 실패).
- 명명: `test_<단위>_<시나리오>_<기대결과>`.
- 한 어서션 = 한 개념. AAA(Arrange/Act/Assert) 배치.

## 5. 보고

다음을 한 곳에 적는다:
- 어떤 명령을 어떤 인자로 돌렸는가
- 통과/실패 카운트
- 새로 추가한 테스트와 그 이유
- 아직 검증 안 된 것 (정직하게)

## 금지
- 통과시키려고 어서션 약화 / 테스트 스킵.
- "플래키"라며 재시도 — 한 번 더 깨질 가능성을 진단한다.
- `2>/dev/null`로 에러 숨기기.
