# 개발 작업 가이드라인

> 이 파일은 프로젝트 루트에 있으며, 모든 Claude Code 세션 시작 시 자동으로 로드됩니다.
> 지시사항은 약 150줄 이내로 유지하세요. **반드시** 매번 지켜져야 하는 것은
> 이 문서가 아니라 `.claude/hooks/`에 넣어야 합니다(이 문서는 권고, hooks는 강제).
> 언어/프로젝트 특수 사항은 `.claude/rules/` 또는 `.claude/skills/`에 넣으세요.

## 0. 첫 실행 (한 번만)

이 가이드라인 세트(`CLAUDE.md` + `.claude/`)를 이 프로젝트에 막 도입했다면,
Claude Code에서 다음 명령을 한 번 실행하세요.

```
/venom            # 표준 흡수 (권장)
/venom deep       # 깊은 흡수 — 기존 rules/skills/hooks까지 이 프로젝트에 맞게 진화
```

`/venom`는 현재 프로젝트를 자동 분석하여 프로젝트 특화 규칙·스킬·훅·메모리를
생성하거나(표준), 기존의 범용 파일까지 이 프로젝트에 맞게 *진화*시킵니다(deep).
자세한 절차는 `.claude/commands/venom.md` 참고.

## 1. 운영 원칙

1. **행동하기 전에 계획한다.** 도구 호출이 2회를 넘는 작업이라면 먼저
   TodoWrite로 짧은 계획을 세운다. 진행하면서 갱신한다.
2. **쓰기 전에 읽는다.** 이번 세션에서 한 번도 읽지 않은 파일은 절대 수정하지
   않는다. 새 파일을 만들기 전에 편집해야 할 기존 파일이 있는지 먼저 확인한다.
3. **작고 되돌릴 수 있는 단계로 진행한다.** 큰 한 방 수정보다 검증을 끼운 작은
   수정 여러 번을 선호한다. 사이사이에 테스트/타입체크/린트를 돌릴 수 있다면 돌린다.
4. **암묵적 가정 금지.** 요구사항이 모호하면 추측하지 말고 핵심 질문 하나를 한다.
5. **저장소 밖으로 나가지 않는다.** 사용자가 명시적으로 요청하지 않는 한
   프로젝트 루트 밖의 파일은 건드리지 않는다. `~`, `/etc`, 시스템 경로는 읽기 전용으로 취급한다.

## 2. 의무 작업 루프

비단순 작업은 모두 이 루프를 따른다.

1. **이해** — 관련 파일을 읽고, 코드베이스를 검색하고, 목표를 자기 말로 다시 정리한다.
2. **계획** — 마지막 단계로 *검증*이 들어간 TodoWrite 목록을 만든다.
3. **구현** — 가장 작게 동작하는 변경부터.
4. **검증** — 프로젝트의 테스트/린트/타입체크 명령을 돌린다. 없다면 최소한
   변경한 파일을 다시 읽고 엣지 케이스를 추론한다.
5. **보고** — 무엇을 바꿨고, 무엇을 검증했고, 무엇을 검증하지 *않았는지* 요약한다.

## 3. 기억(메모리) 프로토콜 — 자기 교정

이 프로젝트는 `.claude/memory/` 아래에 영구 메모리를 둔다.

- **작업 시작 전에** `.claude/memory/mistakes.md`와 `.claude/memory/lessons.md`를
  읽는다(SessionStart hook이 자동으로 컨텍스트에 주입한다). 과거에 저질렀고
  반복하면 안 되는 실수가 들어 있다.
- **실수가 발생했을 때** (테스트 실패, 잘못된 가정, 되돌려진 수정, 사용자 교정)
  `mistake-recorder` 스킬로 `mistakes.md`에 항목을 추가한다. 구체적으로:
  무엇을 했고, 왜 틀렸고, 다음엔 어떻게 해야 하는지.
- **항구적으로 배운 것**(아키텍처, 함정, 관례)은 `lessons.md`에 추가한다.
- **아키텍처적 결정**은 `decisions.md`(ADR 형식)에 남긴다.

메모리 파일은 의도적으로 저장소와 함께 버전 관리된다. 임시 스크래치가 아니다.

## 4. 코드 품질 기본값 (언어 무관)

- 주변 스타일을 따른다. 건드리지 않은 줄을 재포맷하지 않는다.
- 함수는 한 가지 일만 한다. 50줄 또는 3단계 중첩을 넘으면 분리를 고려한다.
- 이름은 구현이 아니라 의도를 드러낸다.
- 에러는 명시적으로 처리한다. 예외를 조용히 삼키지 않는다.
- 죽은 코드, 주석 처리된 블록, 담당자 없는 `TODO` 금지.
- 비밀키, 토큰, API 키, 개인 절대 경로를 소스에 두지 않는다.
- 공개 API와 내보내는 심볼에는 한 줄 설명(주석/도크스트링)을 단다.

## 5. 테스트 기본값

- 버그 수정은 회귀 테스트가 추가될 때까지 끝난 게 아니다.
- 신규 기능은 해피패스 1개와 엣지케이스 1개 이상의 테스트가 있어야 끝난다.
- 프로젝트에 테스트 명령이 있으면 완료를 선언하기 전에 돌린다.
- 테스트가 느리면 타깃 부분집합을 먼저, 그 다음 전체 스위트를 돌린다.

## 6. Git과 커밋

- 사용자가 명시적으로 요청하지 않는 한 절대 커밋하지 않는다.
- 공유 브랜치에 `push --force` 금지.
- `--no-verify`, `--amend`(타인 커밋 대상) 금지.
- 커밋 메시지: 명령형, 제목 ≤ 72자, 본문은 *왜*를 설명한다.
- 파일은 이름으로 명시적으로 스테이징한다. `git add -A` / `git add .` 금지.

## 7. 도구 사용 에티켓

- `cat`/`find`/`grep` 셸 명령보다 Read/Glob/Grep 도구를 우선한다.
- 의존 관계 없는 도구 호출은 가능하면 병렬로 실행한다.
- 오래 걸리는 명령은 백그라운드로.
- sleep 폴링 금지. hook 실패를 무시하지 말 것.

## 8. 일이 잘못됐을 때

hook이 동작을 차단하면 **이유를 읽고 근본 원인을 고친다.** 우회하지 않는다.
정말 hook이 잘못됐다고 판단되면 멈추고 사용자에게 말한다.

## 9. 부록 규칙 파일

`.claude/rules/` 아래 파일은 자동 로드되어 이 문서를 보강한다. CLAUDE.md는
*인덱스*로 두고 세부는 규칙 파일에 둔다.

- `00-core.md` — 협상 불가 안전 규칙
- `10-coding-standards.md` — 포맷, 네이밍, 구조
- `20-security.md` — 비밀키, 의존성, 인젝션
- `30-git-commit.md` — 커밋/브랜치/PR 규약
- `40-testing.md` — 테스트 배치와 기대치
- `50-memory-protocol.md` — `.claude/memory/` 사용법
